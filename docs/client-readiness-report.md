# تقرير جاهزية المشروع للعميل — DressnMore

**التاريخ:** 2026-06-17
**النطاق:** الباك إند `Back-DressnMore` (Laravel 13، multi-tenant SaaS) + الفرونت إند `Front-DressnMore` (React 19 + TypeScript + Vite).
**هدف التقرير:** تقييم جاهزية النظام لتسليم العميل، مع اختبار فعلي للواجهات و الـ API كمستأجر (tenant).

---

## 1) الخلاصة التنفيذية (Verdict)

| المكوّن | الحالة الوظيفية | الجاهزية للإنتاج |
|---|---|---|
| الباك إند (API) | **يعمل بشكل ممتاز** — كل وحدات الأعمال تستجيب بنجاح | **مشروط** — يوجد عوائق أمنية + مجموعة الاختبارات حمراء |
| الفرونت إند (UI) | **واجهة ناضجة وأنيقة** (عربي/RTL) | **غير جاهز** كما هو — لا يتكامل مع هذا الباك إند |
| التكامل Front ↔ Back | **لا يعمل حالياً** | **عائق تسليم حرج** |

**الحكم العام:** الباك إند منتَج قوي ومكتمل وظيفياً على مستوى الـ API ويصلح لمرحلة UAT/Staging بعد معالجة عوائق الأمان. أما **التكامل بين الفرونت والباك فمكسور حالياً** لأنهما مبنيّان على عقد API مختلف (مسارات، ترويسات، CORS، شكل استجابة تسجيل الدخول). **لا يمكن تسليم النظام للعميل كحل متكامل قبل معالجة فجوة التكامل.**

---

## 2) ما الذي تم اختباره فعلياً

### الإعداد
- MariaDB 10.11، PHP 8.3، تهجير القاعدة المركزية + بذور (plans/super-admin/payment-gateways).
- إنشاء مستأجر تجريبي عبر الأمر المدمج `tenant:create-test` ثم بذر بيانات واقعية عبر `demo:tenant-seed`.
- تشغيل خادم Laravel (`php artisan serve`) واختبار الـ API بـ curl كمستأجر مسجَّل الدخول.
- تشغيل الفرونت إند (Vite) وربطه بالباك المحلي واختباره في متصفح Chrome فعلياً.

### نتيجة اختبار الـ API كمستأجر (52/52 نجاح)
سكربت الاختبار: `scripts/readiness_smoke.sh` — غطّى تسجيل الدخول وكل الوحدات الرئيسية:
- المصادقة والهوية: `login`, `me`, `lookups`, `subscription/overview`.
- العمليات: العملاء، الفروع، الفساتين والتصنيفات، الفواتير/المبيعات/الإيجار، التسليم/الإرجاع، التفصيل (tailoring)، ورشة العمل.
- المالية: المصروفات، الصناديق، الحركات النقدية، المدفوعات، المحاسبة وقيود اليومية، دفتر الأستاذ.
- الموردون وأوامر الشراء ومدفوعات الموردين.
- الموارد البشرية (HR) بالكامل: لوحة، أقسام، مسميات وظيفية، موظفون، ورديات، حضور، إجازات، رواتب، إعدادات، وثائق.
- الإشعارات، المصانع، الموظفون.
- اختبارات أمنية: رفض الطلب بدون `X-Tenant` (400)، ورفض التوكن غير الصالح (401).

### نتيجة بوابة ميزات الخطة (Plan Feature Gating) — تعمل بدقّة
- على خطة **Pro**: نقاط `accounting/*` و`supplier-payments` تُرجع **403 "Feature is not available"** (صحيح).
- بعد الترقية إلى **Enterprise**: نفس النقاط تُرجع **200** (صحيح). هذا يثبت أن آلية ربط الميزات بالاشتراك تعمل في الاتجاهين.

## 3) الباك إند — التقييم

### نقاط القوة
- **عزل مستأجرين قوي**: قاعدة بيانات منفصلة لكل مستأجر (`TenantDatabaseManager`)، مع ربط التوكن بـ `tenant_id` ومنع الوصول عبر مستأجر آخر (`EnsureTenantTokenBinding`).
- **بنية نظيفة ومتسقة**: FormRequests للتحقق، طبقة Services، استجابة موحّدة `ApiResponse`، و API Resources.
- **سطح وظيفي واسع جداً** ومكتمل: إيجار/بيع/تفصيل، تسليم/إرجاع/تسوية، مخزون، مصروفات، صناديق، محاسبة وقيود، موردون وأوامر شراء، HR كامل، تقارير، تأمين (security deposit)، انتحال شخصية المستأجر للدعم (impersonation) مع تدوين تدقيقي.
- **بوابات متعددة الطبقات**: اشتراك + صلاحيات (`tenant.permission`) + ميزات الخطة (`plan.feature`).
- **توثيق غني**: `docs/api-contract.md`، `docs/frontend-backend-integration-contract.md`، إلخ.

### عوائق حرجة (Blockers) قبل الإنتاج
1. **مجموعة الاختبارات الآلية حمراء حالياً**: تشغيل `php artisan test` أعطى **263 اختباراً: 68 ناجح، 191 فاشل، 4 أخطاء**.
   - السبب الجذري: الميدلوير `EnsureTenantTokenBinding` غير متوافق مع `Sanctum::actingAs()`. فدالة `actingAs` تُنشئ **Mock** لنموذج `PersonalAccessToken` (وليس `TransientToken`)، وقيمة `tenant_id` عليه `null`، مما يُفعّل الرفض `403 "Token is not valid for this tenant"` في كل الاختبارات التي تعتمد على `actingAs`.
   - الأثر: ملف CI (`.github/workflows/tests.yml`) يشغّل الاختبارات على كل push/PR إلى main ⇒ **خط الـ CI مكسور**، ولا توجد شبكة أمان فعلية ضد الانحدارات.
   - الحل المقترح: إمّا تعديل الاختبارات لإصدار توكنات حقيقية مربوطة بالمستأجر بدل `actingAs`، أو جعل الميدلوير يتعامل مع توكن `actingAs` (مثلاً عبر فحص نوع التوكن/البيئة). الاختبارات التي تستخدم توكنات حقيقية (مثل `HrTenantIsolationTest`) تنجح فعلاً.
2. **لا يوجد Rate Limiting على تسجيل الدخول** (`/api/tenant/login`, `/api/platform/login`) ولا على نقاط التسجيل العامة — تعرّض لهجمات Brute-force.
3. **توكنات Sanctum لا تنتهي صلاحيتها** (`expiration => null`) — لا تدوير ولا انتهاء.

### مخاوف متوسطة
- توكن مدير المنصّة (platform admin) يمكنه الوصول لبعض مسارات المستأجر غير المحميّة بنوع المستخدم (مثل `settings/profile`, `logout`, `me`).
- وحدة HR غير مربوطة بميزات الخطة (متاحة بغض النظر عن مستوى الاشتراك).
- `tenants.plan_id` بدون مفتاح خارجي (FK).
- نقطة الـ health تكشف اسم قاعدة بيانات المستأجر.
- مسار `seed` مكرّر في `routes/api/platform.php`.

### ملاحظات بيئة الإنتاج
- ضبط `APP_DEBUG=false`، تعيين `CORS_ALLOWED_ORIGINS` الحقيقية، كلمة مرور قوية لمدير المنصّة.

## 4) الفرونت إند — التقييم

### نقاط القوة
- **واجهة احترافية**: React 19 + Tailwind 4 + Radix، عربي/RTL، خط Cairo، حالات تحميل/فراغ، ErrorBoundary.
- **تغطية وحدات واسعة**: لوحة تحكم، فواتير/طلبات، مبيعات، تفصيل، عملاء، فساتين، موردون، أوامر شراء، صناديق، محاسبة، موظفون، ورشة، فروع، إشعارات لحظية (Echo/Reverb).
- **بنية API منظّمة**: خدمات `api/v2` + React Query + أنواع TypeScript.

### عوائق حرجة
1. **وحدة الاشتراك (Subscription) بيانات وهمية فقط** (`mocks/subscription.ts`) — غير مربوطة بالخادم، وعليها لافتة صريحة بذلك.
2. **لا توجد أي اختبارات آلية** في الفرونت إند (صفر ملفات اختبار).
3. **لا يوجد زر تسجيل خروج (Logout) ظاهر** للمستخدم (الخروج يحدث فقط عند 401 أو حذف الحساب).
4. **مسارات `sales` و`tailoring` بلا حارس صلاحيات** في الواجهة.

### ملاحظات أخرى
- وحدات موجودة لكنها مخفية من القائمة الجانبية (الصلاحيات/الأدوار، HR الإداري، المخزون — معلّقة في `constants.ts`).
- الصلاحيات تُقرأ من استجابة تسجيل الدخول فقط (لا تتحدث إلا بعد إعادة الدخول).
- استخدام واسع لـ `any` (القاعدة معطّلة في ESLint).

## 5) فجوة التكامل بين الفرونت والباك (العائق الأهم)

عند تشغيل الفرونت إند فعلياً مقابل هذا الباك إند، **لا يتكامل النظام**، للأسباب التالية المثبتة بالاختبار:

1. **عدم تطابق المسارات وأسماء الموارد**
   - الفرونت يستدعي (نسبةً إلى قاعدته الافتراضية `/api/v1`): `/login`, `/clients`, `/cities`, `/currencies`, `/roles`, `/job-titles`, `/transfers`, `/orders/{id}/{status}` …
   - الباك يعرض تحت `/api/tenant`: `/login`, `/customers`, `/lookups`, `/dresses/{id}/transfer`, `/invoices`, `/orders/rental` …
   - أثبتنا بـ curl أن مسارات الفرونت الافتراضية تُرجع **404** على الباك: `POST /api/v1/login → 404`, `POST /api/v1/clients → 404`.
   - (الأسماء مختلفة جذرياً: `clients` مقابل `customers`، ولا وجود لـ `cities/currencies/roles` كنقاط مستقلة في الباك.)

2. **ترويسة `X-Tenant` غير مُرسَلة من الفرونت**
   - الباك يتطلّب `X-Tenant: <slug>` على كل النقاط المصادَّقة (يرفض بـ 400 بدونها — تم التحقق).
   - الفرونت لا يرسلها (يعتمد على نموذج subdomain + استجابة تسجيل الدخول `endpoints.backend_api_url`). أكّد فحص الشبكة في المتصفح **غياب الترويسة**.
   - النتيجة العملية بعد تسجيل دخول ناجح: لوحة التحكم تفشل برسالة **"Tenant context is required" (400)**.

3. **تعارض CORS / withCredentials**
   - أكسيوس في الفرونت يستخدم `withCredentials: true`، بينما `config/cors.php` في الباك فيه `supports_credentials => false`.
   - النتيجة: حظر المتصفح للطلب بـ **CORS error** قبل وصوله للخادم (تم تصويره). ولأن المصادقة قائمة على Bearer Token، فإن `withCredentials` غير ضروري أصلاً.

4. **عدم تطابق شكل استجابة تسجيل الدخول**
   - الفرونت يتوقّع `endpoints{ backend_api_url, reverb_public_url, frontend_app_url }` و`roles` و`account_type`.
   - الباك يُرجع `token, user, tenant, permissions` فقط (بدون `endpoints`). لذلك آلية تحويل قاعدة الـ API بعد الدخول في الفرونت لن تعمل.

> الخلاصة: الفرونت إند يبدو مبنيّاً على عقد API مختلف (غالباً خادم/إصدار آخر منشور على `api.dressnmore.it.com`). لمواءمته مع هذا الباك إند يلزم: توحيد البادئة إلى `/api/tenant`، حقن ترويسة `X-Tenant`، إيقاف `withCredentials` (أو تفعيل credentials في الباك)، ومواءمة أسماء الموارد وشكل استجابة الدخول.

## 6) التوصيات (حسب الأولوية)

**لتسليم متكامل للعميل، يجب أولاً سدّ فجوة التكامل:**
1. الاتفاق على عقد API واحد مجمَّد بين الفريقين (الباك يملك `docs/frontend-backend-integration-contract.md` كنقطة بداية).
2. في الفرونت: ضبط القاعدة على `/api/tenant`، حقن `X-Tenant` في كل الطلبات، إيقاف `withCredentials`، ومواءمة أسماء الموارد (`clients→customers` …) وشكل استجابة الدخول.
3. ربط وحدة الاشتراك في الفرونت بالـ API الحقيقي بدل الـ mock، وإضافة زر تسجيل خروج.

**في الباك إند قبل الإنتاج:**
4. إصلاح توافق `EnsureTenantTokenBinding` مع الاختبارات وإعادة خط الـ CI للأخضر.
5. إضافة Rate Limiting لتسجيل الدخول والنقاط العامة.
6. ضبط انتهاء صلاحية توكنات Sanctum.
7. حصر مسارات المستأجر على مستخدمي المستأجر فقط (منع توكن مدير المنصّة).
8. إعدادات الإنتاج: `APP_DEBUG=false`، CORS، كلمات مرور قوية.
